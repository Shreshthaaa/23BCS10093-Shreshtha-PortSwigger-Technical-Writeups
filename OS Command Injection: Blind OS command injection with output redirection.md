# Title: OS Command Injection: Blind OS command injection with output redirection

## Description

The application is vulnerable to Blind OS Command Injection within the feedback submission functionality. User-controlled input is incorporated into a backend shell command without proper sanitization or validation.

Although command output is not directly returned in the HTTP response, the application allows attackers to redirect command output to a writable directory exposed through the web server. By leveraging output redirection, an attacker can write command results to a publicly accessible file and subsequently retrieve the contents through a browser request.

This vulnerability enables attackers to execute arbitrary operating system commands and disclose sensitive system information.

---

## Steps to Exploit

### 1. Identify the Injection Point

1. Navigate to the feedback page.
2. Submit the feedback form.
3. Intercept the request using Burp Suite.
4. Locate the `email` parameter.

---

### 2. Inject an Operating System Command

Modify the `email` parameter to execute the `whoami` command and redirect its output to a file in the web-accessible directory:

```http
email=||whoami>/var/www/images/output.txt||
```

The payload performs the following actions:

```bash
whoami > /var/www/images/output.txt
```

This executes the `whoami` command and writes the result to a file named `output.txt`.

---

### 3. Retrieve the Command Output

1. Intercept a request for any product image.
2. Locate the `filename` parameter.
3. Replace the original filename with the newly created file:

```http
filename=output.txt
```

4. Forward the request.

---

### 4. Observe the Response

The response contains the contents of the file generated by the injected command.

Example:

```text
peter
```

The returned value represents the operating system user account under which the application is running.

## Impact

* Arbitrary operating system command execution.
* Disclosure of sensitive system information.
* Ability to enumerate users, files, and system configuration.
* Potential exposure of application secrets and credentials.
* Risk of privilege escalation depending on execution context.
* Potential path to full Remote Code Execution (RCE).
* Complete compromise of confidentiality, integrity, and availability of the underlying server.

---

## Mitigation / Remediation

### 1. Avoid Shell Command Execution

Do not invoke shell commands using user-controlled input.

Prefer safe application APIs and native library functions whenever possible.

### 2. Validate and Sanitize Input

Implement strict allow-list validation for all user-supplied input.

Reject shell metacharacters including:

```text
;
&
|
||
$
`
()
<>
>
<
```

### 3. Restrict File System Permissions

Ensure application users cannot write files to publicly accessible web directories.

### 4. Apply Principle of Least Privilege

Run application services using dedicated low-privilege accounts.

### 5. Disable Unnecessary Command Execution

Where command execution is unavoidable, use secure APIs that separate commands from arguments.

### 6. Perform Security Assessments

Conduct periodic penetration testing and code reviews to identify command injection vulnerabilities.

## Conclusion

The application is vulnerable to Blind OS Command Injection within the feedback functionality. By leveraging output redirection, an attacker can write command output to a web-accessible directory and retrieve the results through normal application requests. Successful exploitation enables arbitrary command execution, system reconnaissance, and may ultimately lead to full server compromise.
